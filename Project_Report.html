1.   Introduction

Non-profit organisations receive a large volume of diverse support messages every day — ranging from donation inquiries and volunteer sign-ups to urgent emergency requests and complaints. Manually reading, categorising, and responding to each message is time-consuming, error-prone, and operationally expensive for resource-limited organisations.

The Non-Profit Support Triage Agent is an intelligent, AI-powered system built to automate this process end-to-end. The system accepts an incoming support message through a web interface, analyses it using a pipeline of specialised agents, and produces a structured output that includes the intent classification, urgency level, extracted entities (such as names, emails, phone numbers, and locations), an empathetically drafted AI response, and a routing decision — all within seconds.

The system is built using FastAPI for the backend REST API, Groq's Llama 3.3 70B as the large language model (LLM) for classification and response generation, spaCy for Named Entity Recognition (NER), and a clean, dark-themed HTML/CSS/JavaScript frontend served directly by the backend server.

2.   Project Objectives
  • Automatically classify the intent of incoming support messages into one of five categories: Donation, Volunteer, Complaint, Emergency Help, or General Inquiry.
  • Assess the urgency level (High, Medium, or Low) of each message to prioritise critical cases.
  • Extract structured named entities including full name, phone number, email, location, case ID, and date from unstructured text.
  • Apply deterministic routing rules to route messages to the correct internal team: Human Agent, Finance Team, Volunteer Team, or General Support.
  • Generate a warm, empathetic AI response tailored to the sender's context and needs.
  • Provide a session-aware conversation memory so that multi-turn exchanges within the same session retain context.
  • Serve a real-time web UI (HopeLink Triage) that allows non-technical staff to use the system with a single click.
  • Expose a clean REST API at /triage so the system can be integrated into any external platform.

3.   Proposed Solution

The proposed solution is a Reactive Triage Agent that automates the initial processing of non-profit communications using a modular, multi-agent, LLM-driven architecture. The system accepts incoming messages through a browser-based user interface (HopeLink Triage), processes them via a FastAPI backend, and applies a parallel pipeline of specialised agents to generate structured outputs.

The solution uses a parallel async pipeline to maximise performance: the Classifier Agent and NER Agent run simultaneously in Round 1. Once their outputs are available, the Responder Agent generates an empathetic reply in Round 2 using the full context. The Decision/Routing Agent applies deterministic rule-based logic — requiring no LLM call — making the routing step both instant and fully predictable.

The backend is implemented using FastAPI, which exposes RESTful endpoints for processing requests. The LLM inference is powered by Groq (Llama 3.3 70B) for ultra-fast response times (typically under 5 seconds end-to-end). The NER step combines Groq's LLM extraction with a fallback using spaCy for robust entity detection even on short or poorly-formatted messages.

The frontend is a standalone HTML/CSS/JavaScript single-page application (HopeLink Triage) served directly by the FastAPI server at /ui, with no separate build step or Node.js process required. It features a glass-morphism dark UI with real-time result rendering, example message pills, session ID support, and graceful error handling.

Pipeline Summary:
Round 1 (Parallel): Classifier Agent + NER Agent → 2 concurrent LLM calls
Round 2 (Sequential): Responder Agent → 1 LLM call using Round 1 output
Routing: Decision Agent → Deterministic rules, 0 LLM calls

9.   Conclusion

The Non-Profit Support Triage Agent successfully demonstrates the power of combining modern large language models with deterministic rule-based systems to solve a real-world operational problem faced by non-profit organisations.

By using a parallel async pipeline — where classification and entity extraction run simultaneously — the system achieves significantly lower latency compared to sequential approaches. The integration of Groq's high-speed Llama 3.3 70B model ensures that even complex messages are processed in under five seconds, making the tool practically useful for live customer support scenarios.

The use of deterministic routing rules (no LLM call for the decision step) ensures that routing is predictable, auditable, and never wastes API quota — an important consideration for free-tier deployments. The fallback mechanisms in both the NER pipeline (spaCy hints) and the response generation step ensure robustness even under edge-case inputs.

The system is architected with clean separation of concerns: each agent (classifier, NER, decision, responder) is an independent module with its own prompt template or logic, making it straightforward to extend, test, or swap components. The REST API design enables easy integration with third-party platforms such as CRM systems, email clients, or chatbots.

Overall, this project showcases how a lightweight, production-grade AI application can be built using open source tools and a free-tier LLM API, making intelligent automation accessible even to resource-constrained organisations.

Future Enhancements

  • Integration with a persistent database (PostgreSQL) to store all triage records for analytics and reporting.
  • Email/SMS notification system to automatically forward urgent triage results to the relevant team.
  • Admin dashboard to view historical messages, response times, and routing statistics.
  • Multi-language support using LLM translation as a pre-processing step.
  • Feedback loop: allow agents to correct AI responses, retraining the routing model over time.
  • Deployment to cloud platforms (e.g., Render, Railway, AWS Lambda) for production access.
